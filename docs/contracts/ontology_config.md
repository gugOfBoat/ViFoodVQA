# Ontology Config Contract

> **Canonical source** for the ViFoodKG ontology schema governance.
> The ontology itself lives at `config/ontology_config.json`.

## Purpose

`config/ontology_config.json` defines the complete ontology for the Vietnamese food Knowledge Graph. It specifies entity types, relationship types, question types, and triple extraction requirements. This document describes the schema, its consumers, and the impact of changes.

## Schema Overview

The config contains four top-level sections:

> **Note:** The `entity_classes` array in the config defines **10** types. Two additional node labels — `Condiment` and `SubstitutionRule` — are created by the ingestor at runtime but are not listed in the config.

### 1. `entity_classes` (11 types)

Each entity class defines a node label in Neo4j:

| Entity Class | Description | Example |
|-------------|-------------|---------|
| `Dish` | Complete Vietnamese dish | Phở Bò, Bún Chả |
| `Ingredient` | Individual visible ingredient | Thịt Bò, Tôm |
| `IngredientCategory` | Food group classification | Thịt, Hải sản, Rau lá |
| `Region` | Geographic origin | Hà Nội, Huế, Miền Tây |
| `DishType` | Dish form/category | Món nước, Món khô, Bánh |
| `CookingTechnique` | Cooking method | Ninh, Xào, Chiên, Hấp |
| `FlavorProfile` | Taste characteristic | Cay, Chua, Ngọt, Umami |
| `DietaryTag` | Dietary classification | `animal_product`, `plant_based` |
| `Allergen` | Potential allergen | Giáp xác, Gluten, Đậu phộng |
| `SideDish` | Accompaniment/condiment | Rau Sống, Quẩy, Đồ Chua |
| `SubstitutionRule`* | Reified substitution node | (auto-generated by ingestor) |
| `Condiment`* | Condiment/sauce | Nước Mắm, Tương Ớt |

\* Not listed in `entity_classes` array — created at runtime by `04_neo4j_ingestor.py` and referenced in `01_generate_vqa.py` (`TARGET_LABEL_TO_NODE`).

> **Note:** `SubstitutionRule` is not listed in `entity_classes` but is created by the ingestor during reification of `hasSubRule` triples. `Condiment` exists as a classification category in `02_kg_entity_classifier.py` and a Neo4j label in `04_neo4j_ingestor.py`, but is not formally part of the ontology config.

### 2. `relationships` (12 types)

| Relation | Domain → Range | Hop | Description |
|----------|---------------|-----|-------------|
| `hasIngredient` | Dish → Ingredient | 1 | Dish contains this ingredient |
| `ingredientCategory` | Ingredient → IngredientCategory | 2 | Ingredient belongs to food group |
| `originRegion` | Dish → Region | 1 | Geographic origin |
| `dishType` | Dish → DishType | 1 | Form of dish |
| `servedWith` | Dish → SideDish | 1 | Accompaniment |
| `cookingTechnique` | Dish → CookingTechnique | 1 | Primary cooking method |
| `flavorProfile` | Dish → FlavorProfile | 1 | Taste characteristic |
| `hasAllergen` | Ingredient → Allergen | 2 | Contains allergen |
| `hasDietaryTag` | Ingredient → DietaryTag | 2 | Dietary classification |
| `hasSubRule` | Dish → SubstitutionRule | 1 | Substitution rule (reification) |
| `fromIngredient` | SubstitutionRule → Ingredient | — | Original ingredient |
| `toIngredient` | SubstitutionRule → Ingredient | — | Replacement ingredient |

### 3. `question_types` (11 types)

Maps each question type to its KG relationship path, keywords, and template. These align with entries in `question_types.csv`.

### 4. `triple_extraction_requirements`

Defines mandatory properties for extracted triples:
- `source_url` — URL of the web source
- `evidence` — Textual snippet proving the triple
- `confidence` — `high`, `medium`, or `low`

## Consumers

| Consumer | File | How it uses the ontology |
|----------|------|------------------------|
| Triple extractor | `ViFoodKG/src/03_kg_triple_extractor.py` | Prompt template uses entity classes and relations to guide LLM extraction |
| Neo4j ingestor | `ViFoodKG/src/04_kg_neo4j_ingestor.py` | Creates nodes with labels from entity_classes, edges from relationships |
| Vectorizer | `ViFoodKG/src/05_kg_vectorizer.py` | `VECTORIZE_REL_TYPES` must include all relationship names |
| KG Retriever | `ViFoodVQA/src/query.py` | `_RELATION_TO_VI` mapping must cover all relationships; `_TRAVERSE_QUERY` Cypher must handle all hop patterns |
| VQA generator | `ViFoodVQA/src/01_generate_vqa.py` | `select_candidates()` must have a constructor for each question type; `SUPPORTED_QTYPES` set must match |
| Question types CSV | `*/data/question_types.csv` | Derived from `question_types` section; must stay aligned |

## Change Impact Matrix

### Adding a new entity class

| Step | File | Action |
|------|------|--------|
| 1 | `config/ontology_config.json` | Add entry to `entity_classes` |
| 2 | `ViFoodKG/src/03_kg_triple_extractor.py` | Update prompt template to include new class |
| 3 | `ViFoodKG/src/04_kg_neo4j_ingestor.py` | May need new node creation logic if label differs |
| 4 | Test | Re-extract for a few entities and verify Neo4j |

### Adding a new relationship

| Step | File | Action |
|------|------|--------|
| 1 | `config/ontology_config.json` | Add entry to `relationships` |
| 2 | `ViFoodKG/src/03_kg_triple_extractor.py` | Update extraction prompt |
| 3 | `ViFoodKG/src/04_kg_neo4j_ingestor.py` | Handle new relationship in ingest logic |
| 4 | `ViFoodKG/src/05_kg_vectorizer.py` | Add to `VECTORIZE_REL_TYPES` |
| 5 | `ViFoodVQA/src/query.py` | Add to `_RELATION_TO_VI` mapping; update `_TRAVERSE_QUERY` if hop pattern differs |
| 6 | `ViFoodVQA/src/01_generate_vqa.py` | Add candidate constructor in `select_candidates()` if associated with a new qtype |
| 7 | `question_types.csv` (all 3 copies) | Add row if this creates a new question type |
| 8 | Test | Full pipeline test: extract → ingest → vectorize → retrieve → generate |

### Renaming a relationship

> ⚠️ **High risk.** This is effectively removing + adding.

| Step | File | Action |
|------|------|--------|
| 1 | All files in "Adding a new relationship" | Update to new name |
| 2 | Neo4j | Run migration Cypher to rename existing edges |
| 3 | `kg_triple_catalog` | Update `relation` values in Supabase |
| 4 | `vqa.triples_used` | Update JSONB values in Supabase |
| 5 | All `data/triples/*.json` | Update relation field |

### Adding a new question type

| Step | File | Action |
|------|------|--------|
| 1 | `config/ontology_config.json` | Add entry to `question_types` |
| 2 | `question_types.csv` (all 3 copies) | Add row with canonical_qtype, relationship_path, keywords, descriptions |
| 3 | `ViFoodVQA/src/01_generate_vqa.py` | Add to `SUPPORTED_QTYPES`; add `QTYPE_ALIASES` mapping; implement candidate constructor in `select_candidates()` |
| 4 | Verify | KG must have edges that match the relationship path |
| 5 | Test | `python src/01_generate_vqa.py --qtypes new_qtype --limit-images 5` |

## Versioning

Currently, the ontology has no formal versioning. Changes are tracked via git history.

**Recommended practice:**
- Treat `config/ontology_config.json` as immutable during a VQA generation run
- Document ontology changes in commit messages with `[ontology]` prefix
- After ontology changes, re-run: vectorizer → VQA generator (for affected qtypes)

## Validation Checklist

Before committing ontology changes:

- [ ] `config/ontology_config.json` is valid JSON
- [ ] All entity class names are unique
- [ ] All relationship names are unique
- [ ] All relationship domain/range reference existing entity classes
- [ ] `question_types` entries reference valid relationship paths
- [ ] `VECTORIZE_REL_TYPES` in `05_kg_vectorizer.py` includes all relationship names
- [ ] `_RELATION_TO_VI` in `query.py` includes all relationship names
- [ ] `SUPPORTED_QTYPES` in `01_generate_vqa.py` includes all supported question types
- [ ] `question_types.csv` (all 3 copies) matches the `question_types` section
